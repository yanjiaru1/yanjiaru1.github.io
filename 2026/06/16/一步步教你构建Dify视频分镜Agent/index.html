2. 整体架构

为什么不在 Dify 里直接处理？

Dify 的沙箱（gVisor）有几条硬限制，导致视频处理走不通：

限制 影响
禁止 subprocess 无法运行 FFmpeg
禁止写临时文件 tempfile 权限不足
变量 200KB 上限 base64 图片会被截断
迭代节点变量作用域隔离 迭代内部结果外部拿不到

解决思路： 把「视频下载 + FFmpeg 处理 + AI 分析 + 飞书写入」全部抽到一个独立的 Docker 容器（ffmpeg-server）中，Dify 只负责发起 HTTP 调用和展示结果。

  • 一键还原工作流：导入后自动生成所有节点
  • 版本管理：可以用 Git 管理工作流版本
  • 跨实例迁移：从一个 Dify 实例导出，导入另一个

8. 常见错误与修复

8.1 http_boddy 参数名拼写错误

错误信息：

原因： 代码执行节点配置的输入变量名为 http_boddy（多了一个 d），但 main 函数的参数名是 http_body。沙箱调用 main(http_boddy=...) 时报错。

修复： 在代码节点配置中将变量名改回 http_body，或直接修改数据库：

8.2 代码节点 IndentationError: unexpected indent

原因： 代码执行节点残留了默认模板代码（def main(arg1, arg2) 未删除），且新增的 def main 前面有多余空格缩进。

修复： 删除残留模板代码，def main 必须顶格写（无前导空格）。

8.3 嵌套函数 _fmt_time 缩进错误

原因： def _fmt_time(t) 没有缩进在 main 函数内部，导致 IndentationError。

修复： def _fmt_time(t): 必须缩进在 main 函数体内（7 个空格或 1 个 tab）。

8.4 HTTP 节点 400 Bad Request（三花括号问题）

错误信息：

原因： Dify 的 HTTP 请求节点不支持 {{{...}}}（三花括号）语法，只识别 {{#...#}}（双花括号）。如果代码节点做了 json.dumps(shots) 得到一个字符串，再用 {{{...}}} 插入，最终 body 中包含字面的 {{{` 文本，不是合法 JSON。 **正确做法：** 1. 代码节点输出 `shots` 作为 `array[object]`（原始 Python 列表），不做 `json.dumps` 2. HTTP 节点的 Body 类型选 **json**（非 raw-text） 3. 模板引用使用 `{{#node_id.shots#}}（单层花括号）

8.5 输出类型枚举值错误

错误信息：

原因： 代码节点输出 schema 中 shots 的类型写成了 array_object，但 Dify 的有效枚举值是 array[object]（带方括号）。

修复： array_object → array[object]

8.6 HTTP 超时

原因： 视频太长或分镜太多，处理时间超过默认 60 秒超时。

修复： 将 HTTP 节点的「最大读取超时」和「最大写入超时」设为 600 秒。

8.7 飞书记录重复

原因： Dify 底层在网络波动时会隐式重试 HTTP 请求。

修复： 确保调用的是 http://ffmpeg-server:5002/feishu-create（带去重），而不是直连飞书 API。

8.8 图片在 Dify 输出中不显示

原因： image_url 是容器内地址，浏览器无法直接访问。

修复： 在 nginx 中配置 /shots 代理（见 5.4 节）并重启 nginx。

8.9 ffmpeg-server 日志显示「下载视频失败: HTTP 404」

原因： Dify 文件签名 URL 在容器间无法验证，或 STORAGE_TYPE 不是 local。

修复：

  • 确认 .env 中 STORAGE_TYPE=local
  • 确认 ffmpeg-server 挂载了 ../storage:/app/storage:ro
  • 重启 Dify

关键技术要点

要点 说明
Dify 沙箱变量传递 沙箱调用 main(**inputs_obj)，参数名必须与节点变量配置完全一致
Dify 模板语法 HTTP 节点仅支持 {{#node_id.field#}}，不支持 {{{...}}}
json vs raw-text Body json 类型做 JSON 感知替换（数组/对象 → 原始 JSON）；raw-text 只做文本替换
代码节点输出 直接返回 Python 对象（列表、字典），Dify 自动序列化，无需 json.dumps()
数据库修改 workflow 配置存储在 PostgreSQL 的 workflows.graph 字段中
